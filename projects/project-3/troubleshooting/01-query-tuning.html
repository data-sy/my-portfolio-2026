쿼리 튜닝을 통한 맞춤 문항 API 성능 78.53% 개선

📍 문제 상황

'맞춤 문항 제공 API'에 500명 부하 테스트를 실시했을 때:

  • p(95) 기준 응답 시간: 232.77ms
  • 사용자 경험 목표치(150ms 이하)를 크게 초과
  • DB 쿼리가 병목 지점으로 확인됨

🔧 해결 과정

📊 분석

EXPLAIN과 EXPLAIN ANALYZE를 사용하여 쿼리 실행 계획 분석

  • Seq Scan (순차 스캔)이 발생하고 있음을 확인
  • ORDER BY RAND()가 전체 테이블을 읽은 후 정렬하는 구조
🔴 기존 쿼리
SELECT * FROM questions
WHERE concept_id = ?
ORDER BY RAND()
LIMIT 1;
232.77ms

전체 테이블 스캔 + 랜덤 정렬 발생

↓
시도 1: 모든 데이터 조회 후 Java에서 랜덤 선택
// 모든 데이터를 가져와서 Java에서 랜덤 선택
List<Question> questions = repository.findByConceptId(conceptId);
Question random = questions.get(new Random().nextInt(questions.size()));
333.62ms
❌ 악화 (43.32% 증가)

원인: 데이터 전송량 증가로 오히려 성능 저하

↓
시도 2: ID만 조회 후 Java에서 랜덤 선택
// ID만 조회하여 전송량 최소화
List<Long> ids = repository.findIdsByConceptId(conceptId);
Long randomId = ids.get(new Random().nextInt(ids.size()));
Question question = repository.findById(randomId);
152.6ms
✅ 개선 (34.44% 감소)

개선 포인트: 데이터 전송량 감소, 그러나 쿼리 2회 실행

↓
시도 3: ORDER BY RAND()를 인라인 뷰로 최적화
SELECT q.* FROM questions q
INNER JOIN (
    SELECT id FROM questions
    WHERE concept_id = ?
    ORDER BY RAND()
    LIMIT 1
) AS rand_q ON q.id = rand_q.id;
49.98ms
✅ 개선 (78.53% 감소)

개선 포인트: 인라인 뷰에서 ID만 랜덤 선택 후 조인, DB 레벨에서 최적화

💡 배운 점

  • 쿼리 실행 계획 분석의 중요성: EXPLAIN을 통해 실제 DB가 어떻게 동작하는지 이해하고, 병목 지점을 정확히 파악할 수 있었습니다.
  • "왜 빠른지" 이해하기: 각 시도마다 성능이 개선되거나 악화된 이유를 분석하면서, DB 레벨에서의 효율적인 쿼리 작성 원리를 배웠습니다.
  • 애플리케이션 vs DB 최적화: 때로는 Java에서 처리하는 것보다 DB에서 처리하는 것이 효율적입니다. 데이터 전송량과 쿼리 횟수를 함께 고려해야 합니다.
  • 측정 가능한 개선: p(95) 응답 시간이라는 정량적 지표를 기준으로 개선 효과를 명확히 측정하는 것의 중요성을 배웠습니다.
