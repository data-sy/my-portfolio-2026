CI/CD 자동화로 배포 시간 72% 단축

📍 문제 상황

수작업 배포로 인한 문제:

  • 배포 시간: 25분 이상
  • 컨테이너화 없이 직접 서버에 배포
  • 휴먼 에러 빈번 발생 (설정 누락, 순서 실수 등)
  • 배포 시마다 스트레스와 불안감

🔧 해결 과정

🔴 기존 배포 프로세스
  1. 로컬에서 빌드
  2. SCP로 서버에 파일 전송
  3. SSH 접속 후 기존 프로세스 종료
  4. 환경 변수 설정 확인
  5. 새 프로세스 실행
  6. 로그 확인하여 정상 동작 검증
25분 + 휴먼 에러 위험

시도 1: GitHub Actions로 CI/CD 파이프라인 자동화
# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Build with Gradle
        run: ./gradlew build
      - name: Deploy to EC2
        run: |
          # SSH로 배포 스크립트 실행
✅ 자동화 성공

개선: push만 하면 자동 배포, 휴먼 에러 제거

시도 2: Docker Compose로 다중 서비스 관리
# docker-compose.yml
version: '3.8'
services:
  backend:
    build: ./backend
    ports:
      - "8080:8080"
    depends_on:
      - mysql
      - redis

  mysql:
    image: mysql:8
    volumes:
      - mysql_data:/var/lib/mysql

  redis:
    image: redis:7-alpine

  neo4j:
    image: neo4j:5
    ports:
      - "7474:7474"
✅ 컨테이너화 성공

개선: 단일 EC2에서 여러 서비스를 격리된 환경으로 관리

추가 성과:

  • 휴먼 에러 완전 제거
  • 배포 스트레스 해소
  • 롤백 용이 (이전 이미지로 즉시 복구)
  • 환경 일관성 확보 (개발/운영 동일 환경)

💡 배운 점

  • 자동화의 가치: 반복 작업을 자동화하면 시간 절약뿐 아니라 휴먼 에러 제거와 심리적 안정감까지 얻을 수 있습니다.
  • 컨테이너화의 장점: Docker를 통해 환경 의존성 문제를 해결하고, 개발/운영 환경의 일관성을 확보할 수 있었습니다.
  • 단일 서버에서의 Docker Compose: 비용 제약으로 단일 EC2를 사용해야 할 때, Docker Compose로 여러 서비스를 격리하여 관리할 수 있습니다.
  • 점진적 개선: 처음부터 완벽한 인프라를 구축하기보다, 문제를 느낄 때마다 점진적으로 개선하는 것이 현실적입니다.
