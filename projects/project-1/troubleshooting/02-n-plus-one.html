N+1 문제 해결로 주문 상세 조회 성능 81% 개선

📍 문제 상황

주문 상세 조회 API에서 N+1 문제 발생:

  • 쿼리 수: 21개 (1 + N)
  • 응답 시간: 800ms
  • Order → OrderItem → Product 연관 엔티티 조회에서 병목

🔧 해결 과정

📊 분석

Hibernate SQL 로그를 활성화하여 쿼리 분석

  • Order 조회 1회 + OrderItem 조회 N회 발생
  • 각 OrderItem마다 Product를 다시 조회 (지연 로딩)
  • DB 라운드트립이 병목의 주요 원인
🔴 기존 상태
// 기본 JPA 조회 (지연 로딩)
Order order = orderRepository.findById(orderId);
// 이후 order.getOrderItems() 접근 시 N개 쿼리 추가 발생
21개 쿼리, 800ms
↓
시도 1: @EntityGraph 사용
@EntityGraph(attributePaths = {"orderItems", "orderItems.product"})
Optional<Order> findWithItemsById(Long id);
3개 쿼리, 650ms
✅ 개선 (19% 감소)

문제점: 일부 연관 엔티티에서 여전히 지연 로딩 발생

↓
시도 2: JPQL Fetch Join 적용
@Query("SELECT o FROM Order o " +
       "JOIN FETCH o.orderItems oi " +
       "JOIN FETCH oi.product " +
       "WHERE o.id = :orderId")
Optional<Order> findWithAllDetailsById(@Param("orderId") Long orderId);
1개 쿼리, 600ms
✅ 개선 (25% 감소)

문제점: 쿼리 1개로 줄었으나 데이터 전송량이 증가 (Cartesian Product)

↓
시도 3: Fetch Join + 반정규화 전략 병행
// 1. 필요한 필드만 Projection으로 조회
@Query("SELECT new com.example.dto.OrderDetailDto(" +
       "o.id, o.status, oi.quantity, p.name, p.price) " +
       "FROM Order o " +
       "JOIN o.orderItems oi " +
       "JOIN oi.product p " +
       "WHERE o.id = :orderId")
List<OrderDetailDto> findOrderDetails(@Param("orderId") Long orderId);

// 2. 자주 조회되는 합계는 Order 테이블에 반정규화
// totalAmount, itemCount 컬럼 추가
1개 쿼리, 150ms
✅ 개선 (81% 감소)

개선 포인트: 필요한 데이터만 조회 + 반복 계산 제거

💡 배운 점

  • N+1 해결이 목표가 아님: 쿼리 수를 줄이는 것보다 전체 응답 시간 단축이 진짜 목표입니다. 쿼리 1개로 줄여도 데이터 전송량이 증가하면 효과가 제한적입니다.
  • ORM의 편의성과 비용: JPA 지연 로딩은 편리하지만, 실제 실행 쿼리 수와 I/O 비용을 반드시 확인해야 합니다.
  • Projection의 중요성: 엔티티 전체를 가져오기보다 필요한 필드만 DTO로 조회하면 데이터 전송량을 크게 줄일 수 있습니다.
  • 반정규화 고려: 읽기 빈도가 높은 집계 데이터는 반정규화를 통해 계산 비용을 줄일 수 있습니다.
