# 프로젝트 5: Plogging Community

## 기본 정보
- 프로젝트 이름: 플로깅 모여라!
- 영문 이름: plogging-community
- 버전: v1.0.0
- 기간: 2021.07 ~ 08 (3주)
- 역할: BE 70%, FE 20% (4인 개발: BE 2명, FE 2명)
- 기술 스택: Java 8, JSP & Servlet, Oracle 11g, JavaScript, Kakao Maps API

## 프로젝트 개요
- 서비스 설명: 플로깅 장소를 공유하고 커뮤니티를 구축하여 플로깅 활성화를 도모하는 웹 프로젝트
- 개발 배경: 플로깅(조깅하면서 쓰레기 줍기) 활동을 함께할 사람을 찾고, 좋은 장소를 공유할 수 있는 커뮤니티 필요
- 주요 기능: 플로깅 장소 등록 및 조회, 반경 내 장소 검색, 커뮤니티 게시판

## 정량적 성과
- 성과 1: '반경 내 데이터 조회' 기능 오차율 88.47% 개선 (32.02% → 3.69%)
- 성과 2: 백엔드 시스템 설계 및 50% 구현
- 성과 3: 핵심 비즈니스 로직 개발 (반경 내 데이터 조회, 기준 충족 시 자동 등록)

## 인사이트
- 배운 점 1: 성능과 정확도 모두 중요하지만, 요구사항에 따라 어느 한쪽을 우선해야 할 때가 있음
- 배운 점 2: DBMS에서 제공하는 공간 데이터 관리 기능(PostGIS 등)을 알게 되어, 향후 공간 데이터를 다룰 때 활용할 계획

## 트러블슈팅 목록

### 트러블슈팅 1 (Type: A)
- 제목: '반경 내 데이터 조회' 기능 오차율 88.47% 개선
- 문제 상황: 500m 반경 내 데이터를 조회하는 기능 개발 과정에서 거리 계산 오차 발생
- 해결 과정:
  - 시도 1: 위도/경도 기준 BETWEEN 키워드 사용 → 반경의 의미가 왜곡됨 (사각형 범위)
  - 시도 2: POWER 함수로 원의 반경 계산 → 평면 거리 개념에 머무름 (지구 곡률 미반영)
  - 시도 3: ACOS, COS, SIN 함수로 구면 기하학 거리 계산 → 부동 소수점 오차 발생
  - 시도 4: 구면 기하학 기반 거리 계산을 Java 애플리케이션에서 직접 구현
- 결과: 오차율 32.02% → 3.69% (88.47% 개선)
- 배운 점: 성능보다 정확도가 중요한 상황에서는 애플리케이션 레벨에서 정밀한 계산을 수행하는 것이 적절할 수 있음. 이후 PostGIS 같은 공간 데이터 전문 도구의 존재를 알게 됨
