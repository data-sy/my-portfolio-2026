# 프로젝트 3: MyMathTeacher

## 기본 정보
- 프로젝트 이름: MyMathTeacher
- 영문 이름: my-math-teacher
- 버전: v2 (진행 중)
- 기간:
  - v1: 2024.01 ~ 07 (6개월)
  - v2: 2026.02 ~ (예정)
- 역할: 1인 개발 (기획, 설계, 백엔드, 프론트엔드, AI, 인프라 전체)
- 기술 스택: Java 17, Spring Boot 3, Spring Security 6, JdbcTemplate → Spring Data JPA (리팩토링 중), MySQL 8, Neo4j 5, Redis, Docker, GitHub Actions, AWS EC2, Vue 3, Python, TensorFlow 2

## 프로젝트 개요
- 서비스 설명: 틀린 문제에서 부족한 선수지식을 역추적하는 수학 진단 웹서비스. 학생이 틀린 개념을 기반으로 어떤 선수지식이 부족한지 그래프로 시각화하여 맞춤 학습을 제공
- 개발 배경: 수학 강사로 일하며 "이 학생이 왜 이 문제를 못 풀까?"를 진단하는 도구가 없어 불편함을 느낌. 선수지식 간 연결 관계를 시각화하고, AI로 취약점을 진단하는 서비스를 직접 개발
- 주요 기능: 수학 개념 간 선/후수 관계 그래프 시각화, AI 기반 취약 개념 진단 (AUC 0.83), 맞춤 문항 제공, 학습 이력 관리

## 정량적 성과
- 성과 1: 쿼리 튜닝을 통한 API 성능 78.53% 개선 (232ms → 50ms)
- 성과 2: CI/CD 자동화로 배포 시간 72% 단축 (25분 → 7분)
- 성과 3: Graph DBMS 도입으로 새로운 요구사항에 즉각 대응 가능한 구조 확보

## 인사이트
- 배운 점 1: 쿼리 성능 개선을 위해 EXPLAIN으로 내부 실행 계획을 이해하고 활용해야 함
- 배운 점 2: 기술 선택 시 문제 해결에 필요한 요구사항과 데이터 모델을 충분히 고려해야 함 (RDB vs Graph DB)
- 배운 점 3: 자동화 배포와 Docker 컨테이너화를 통해 운영 안정성을 높이고 확장 기반을 마련할 수 있음

## 트러블슈팅 목록

### 트러블슈팅 1 (Type: A)
- 제목: 쿼리 튜닝을 통한 맞춤 문항 API 성능 78.53% 개선
- 문제 상황: '맞춤 문항 제공 API'에 500명 부하 테스트 진행, p(95) 기준 응답 시간 232.77ms
- 해결 과정:
  - 시도 1: `ORDER BY RAND() LIMIT 1` 사용 → 전체 테이블 스캔 발생
  - 시도 2: 모든 데이터 조회 후 Java에서 랜덤 선택 → 333.62ms (오히려 악화)
  - 시도 3: ID만 조회 후 Java에서 랜덤 선택 → 152.6ms (34% 개선)
  - 시도 4: `ORDER BY RAND() LIMIT 1`를 인라인 뷰로 이동하여 최적화
- 결과: p(95) 기준 응답 시간 232.77ms → 49.98ms (78.53% 개선)
- 배운 점: 성능 최적화를 위해 EXPLAIN으로 실행 계획을 확인하고, 데이터베이스 레벨에서 효율적인 쿼리를 작성해야 함

### 트러블슈팅 2 (Type: A)
- 제목: CI/CD 자동화로 배포 시간 72% 단축
- 문제 상황: 컨테이너화 없이 수작업 배포로 인해 배포 시간이 길고 휴먼 에러 발생
- 해결 과정:
  - 시도 1: GitHub Actions로 CI/CD 파이프라인 자동화
  - 시도 2: 여러 서비스를 단일 EC2에 배포해야 해서 Docker Compose 도입
- 결과: 배포 시간 25분 → 7분 (72% 단축), 휴먼 에러 제거
- 배운 점: 자동화 배포와 컨테이너화를 통해 운영 안정성을 높이고, 서비스 확장에 유연하게 대응할 수 있는 기반 마련

### 트러블슈팅 3 (Type: A)
- 제목: Graph DBMS 도입을 통한 개발 생산성 향상
- 문제 상황: 그래프 데이터를 RDB에서 처리할 때 쿼리 작성에 시간이 과도하게 소요됨
- 해결 과정:
  - 시도 1: 재귀 CTE 문법 활용 → 쿼리 속도 20ms → 2ms (90% 개선), 그러나 여전히 쿼리 복잡도 높음
  - 시도 2: 데이터 모델에 맞는 Graph DBMS(Neo4j) 도입 → 쿼리 복잡도 대폭 감소
- 결과: "유저의 학년에 따라 선수지식 깊이를 다르게 설정"하는 새로운 요구사항에 즉각 대응 가능
- 배운 점: 기술 선택 시 성능뿐 아니라 데이터 모델과 요구사항 변화에 대한 유연성을 함께 고려해야 함
