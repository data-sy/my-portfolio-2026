선수지식 그래프를 어떻게 처리할 것인가?
Neo4j 유지
MySQL 마이그레이션
↓
1 실제 쿼리 패턴은 무엇인가?
📊 분석 결과: "특정 노드의 하위 선수지식 탐색"만 사용
→ 단방향 트리 구조로, 다방향 탐색이나 최단 경로 같은 그래프 특화 쿼리는 미사용
✅ MySQL 8.0 재귀 CTE로 동일하게 구현 가능
↓
2 인프라 관리 부담은 어떤가?
📦 현재 상태: MySQL + Neo4j 이중 관리
→ 두 DB의 설정, 백업, 모니터링을 각각 수행해야 함
✅ MySQL 단일 통합으로 관리 포인트 절반 감소
↓
3 데이터 동기화 문제는?
🔄 현재: 개념(단원) 테이블이 Neo4j와 MySQL 양쪽에 존재
→ 개념 추가/수정 시 양쪽 동기화 필요, 데이터 불일치 리스크
✅ MySQL 단일 소스로 동기화 문제 자체 제거
↓
4 1인 개발 환경에서 운영 가능한가?
👨‍💻 제약: Neo4j 운영 학습 + 장애 대응을 혼자 감당
→ 이미 익숙한 MySQL만 사용하면 관리 부담 최소화
✅ MySQL 선택이 운영 리스크 감소
↓
5 향후 그래프 쿼리 확장 가능성은?
🔮 평가: 현재 요구사항에는 복잡한 그래프 쿼리 없음
→ Neo4j의 핵심 강점(추천, 최단 경로)을 활용하지 못함
⚠️ 그래프 특화 쿼리 포기. 필요 시 재도입 고려
↓
6 마이그레이션 비용은 감당 가능한가?
💰 소요: 데이터 변환 + 재귀 CTE 재구현 + 결과 검증
→ 단기적 비용 발생하나, 장기적 운영 부담 감소가 더 큰 이점
⚠️ 단기 비용 < 장기 이득
↓
