| key | value |
|---|---|
| title | 배치 선집계로 통계 조회 실시간 부하 제거 |
| situation | 통계 조회 시 매번 GROUP BY 실시간 연산으로 DB 부하 발생. 대시보드 조회마다 대량 데이터 집계 |
| option_1_name | 실시간 GROUP BY 조회 |
| option_1_pros | 항상 최신 데이터, 구현 단순 |
| option_1_cons | 데이터 증가 시 응답시간 급증, 동시 조회 시 DB 부하 집중 |
| option_2_name | 캐시 기반 집계 (Redis 캐싱) |
| option_2_pros | 반복 조회 시 빠른 응답, DB 부하 분산 |
| option_2_cons | 캐시 미스 시 실시간 집계 발생, TTL 관리 복잡성, 실시간성 저하 |
| option_3_name | 배치 선집계 (집계 테이블) |
| option_3_pros | 조회 시 단순 SELECT, DB 부하 완전 제거, 일정한 응답속도 |
| option_3_cons | 데이터 실시간성 포기 (배치 주기만큼 지연), 집계 테이블 관리 필요 |
| final_choice | 배치 선집계 (집계 테이블) |
| reasoning | 통계 대시보드는 분/시간 단위 정확도면 충분. 실시간성을 포기하는 대신 조회 성능과 DB 안정성 확보. 배치 주기(5분)로 허용 가능한 수준의 지연 |
| insight_1 | "실시간이어야 하는가?"를 먼저 질문해야 함. 통계/리포트는 대부분 분 단위 지연이 허용되며, 불필요한 실시간 집계는 DB 자원 낭비 |
| insight_2 | 선집계는 "쓰기 시점에 비용을 지불하고, 읽기 시점에 비용을 절약"하는 전략. 읽기 비율이 높은 통계 조회에 적합한 트레이드오프 |
| followup_q1 | 배치 선집계 중 원본 데이터가 변경되면 정합성은? |
| followup_q2 | 실시간 통계가 정말 필요한 경우에는 어떻게 하나? |
