## 지원자 실제 배경 (답안 작성용 컨텍스트 — 면접관은 이 정보를 모름)
- 수학 강사 경력 약 5년 (임용시험에는 합격하지 못함. 학원 수학 강사, 학교 시간강사로 근무)
- 임용시험 준비 기간: 1년 일하며 자금 마련 → 1년 임용 준비, 이 사이클을 반복하며 20대 대부분을 보냄
- 전환 계기: 학생들을 가르치며 "이런 서비스가 있으면 좋겠다"는 아이디어가 생겨 1인 개발에 뛰어듦. 7개월 개발 + 5개월 운영 시도했으나 실서비스화에 실패. 이 경험에서 "개발만으로는 안 되고, IT 서비스의 전체 사이클(기획·개발·운영)을 회사에서 경험해야 한다"는 것을 깨달아 취업을 결심
- 이력서 서사 톤: 수학적 모델링과 백엔드 설계의 유사성에 몰입 → 트레이드오프 판단이 엔지니어링의 본질이라는 것을 체득 → ADR로 의사결정 근거를 기록하는 개발자로 성장 중
- 경력 2~3년 공고에 실무 경험 없이 지원하여 서류 통과함. 프로젝트 경험을 경력으로 인정받은 것으로 추정
- 수학 강사 경험에서 체득한 사고방식: 학생의 오답 패턴 진단 → 근본 원인 파악 → 맞춤 처방. 이것이 디버깅/트러블슈팅 접근과 동일한 구조
- AI 활용 개발 워크플로우를 적극적으로 사용 중 (Claude, Figma 자동화 등)
- ADR(Architecture Decision Record) 문서화 습관
- ⚠️ 창업 의지(이 서비스를 다시 디벨롭하고 싶다)는 면접에서 언급하지 않는다. "서비스 전체 사이클을 경험하고 싶다"까지만 말할 것

## 답안 작성 원칙

### 톤
- 면접에서 실제로 말할 수 있는 구어체 (글로 쓰는 톤 아님)
- **두괄식**: 핵심 결론 먼저 → 근거/경험 → 해당되면 트레이드오프 언급
- 한 답변당 말하기 기준 30초~1분 분량 (3~7문장)
- 자기소개만 예외적으로 1분 분량
- 단, "전체 과정을 설명해주세요" 같은 질문은 1분 30초까지 허용

### 카테고리별 답안 스타일

**A. 프로젝트 기반 질문**
- 반드시 해당 md 파일의 실제 데이터(수치, 시도 과정, 코드 맥락)를 근거로 작성
- "~했습니다"가 아니라 "~해서 p95 70.6ms에서 34.3ms로 51% 개선했습니다" 처럼 구체적 수치 포함
- 꼬리질문 답안은 앞 질문의 답변을 전제하고 이어지도록 작성

**B. CS 기본 질문**
- 개념 설명 + 포트폴리오 경험 연결 (연결 가능한 경우)
- "교과서적 정의"만 말하지 말고, "저는 실제로 이 프로젝트에서 ~할 때 이 개념을 적용했습니다" 식으로 마무리
- 오토위니 연결 포인트가 있는 질문은 "오토위니에서도 ~한 상황에서 유용할 것 같습니다" 식으로 자연스럽게 연결

**C. 서사/컬처핏 질문**
- 임용시험 관련 질문: 솔직하되 구구절절하지 않게. "가르치는 일을 하며 서비스 아이디어가 생겼고, 직접 만들어보다가 개발로 전환했다"는 흐름으로. 임용 미합격 자체에 머무르지 말고 바로 전환 계기로 연결
- 압박 질문: 방어적이지 않고 담담하게. 약점을 인정하되 바로 강점으로 전환
- "왜 오토위니?" 질문: JD 원문의 구체적 내용(DevOps 정착 중, CodeRabbit, 대용량 실시간 처리, 업무 전환 가능)을 인용하며 답변
- 경력 2~3년 공고에 실무 경험 없이 지원한 부분: "경력은 부족하지만 프로젝트에서 실제 성능 문제를 진단하고 해결한 경험이 있다"는 톤으로
