## 1. JD 분석

### ① 표면 요구사항 vs 진짜 니즈

**표면 요구사항**과 **그 뒤에 있는 진짜 니즈**를 항목별로 해석합니다.

**"Java, Spring Framework 기반 실무 경력 2~3년"**
→ 진짜 니즈: 레거시 코드를 읽고 유지보수할 수 있는 사람. 오토위니는 Spring 5.x (Boot 아님)를 쓰고 있고, Oracle 19c + JSP가 있다는 게 힌트입니다. 신규 서비스는 Spring Boot로 가고 있지만, 기존 플랫폼 레거시 코드에 바로 투입될 수 있는 사람이 필요합니다. "실무 경력"을 강조하는 건 학습용 토이 프로젝트가 아닌 실제 트래픽/운영 맥락을 아는 사람을 원한다는 뜻입니다.

**"주도적으로 백엔드 아키텍처를 설계해본 경험"**
→ 진짜 니즈: 주어진 티켓만 소화하는 게 아니라, 문제를 스스로 발견하고 개선 방향을 제안할 수 있는 사람. 고경력 JD에도 똑같이 이 문구가 있다는 건, 팀에 능동적 기여자가 부족하거나 DevOps 전환 과정에서 주도적인 구성원이 필요하다는 신호입니다.

**"대용량 데이터를 실시간으로 처리하는 아키텍처"**
→ 진짜 니즈: 전 세계 100개국 바이어가 접속하는 플랫폼이고, 옥션위니(경매)는 특성상 순간 집중 트래픽이 발생합니다. 경매 입찰 시 실시간 랭킹, 동시 접속 처리 같은 문제가 현재 진짜 페인포인트일 가능성이 높습니다. Redis가 스택에 명시된 것도 이 맥락입니다.

**"JSP 코드 작성 및 수정 가능"**
→ 진짜 니즈: 이건 진짜 레거시 유지보수 수요입니다. 오래된 JSP 기반 화면이 현재 운영 중이고, 그걸 당장 전면 재작성할 리소스가 없는 상황입니다. 전혀 모르면 탈락 요인이 되고, 조금이라도 알면 플러스입니다.

**"JavaScript 코드 작성 및 유지보수 가능"**
→ 진짜 니즈: 풀스택에 가까운 기여를 기대합니다. 프론트 전담 개발자가 있긴 하겠지만, 백엔드 개발자도 간단한 JS 수정 정도는 스스로 처리하길 원합니다. 1차 코테에 JS가 2문제인 것도 이 때문입니다.

**"코드 리뷰, DevOps 문화 정착"**
→ 진짜 니즈: 아직 완성된 조직이 아니고 문화를 만들어가는 단계입니다. CodeRabbit을 쓰고 있다는 건 AI-assisted 개발에 거부감이 없는 팀이라는 뜻이기도 합니다. 코드 품질에 관심 있고, 문화 형성에 기여할 수 있는 사람을 원합니다.

### ③ 기술 스택 매핑

**바로 어필 가능 (겹치는 부분)**

- Java 17, Spring Boot 3, Spring Security, JPA → 오토위니 백엔드 스택과 직접 연결
- MySQL, Redis → 동일
- Docker, GitHub Actions, AWS EC2/RDS → DevOps 문화 정착 니즈에 부합
- JavaScript (Vue 3 사용 경험) → JS 유지보수 가능 조건 충족 증거

**솔직하게 전략적으로 답해야 할 부분**

- **Oracle 19c**: 경험 없음. "MySQL과 SQL 표준 문법은 동일하게 적용했으며, Oracle 특유의 함수 차이는 학습 중입니다"라고 답하면 됩니다. 면접관도 신입에게 Oracle 숙련도를 기대하지 않습니다.
- **Spring 5.x (non-Boot)**: Spring Boot만 써봤지만, 이건 장점으로 전환 가능합니다. "Boot가 자동으로 처리해주는 설정을 의식적으로 이해하려 했고, 그래서 Spring 5.x 코드를 보더라도 Bean 설정, DispatcherServlet 구조는 익숙합니다"라고 하면 됩니다.
- **JSP**: Vue 3를 사용했고 프론트엔드 흐름은 알지만, JSP 직접 작성 경험은 없습니다. "Servlet 기반의 요청 처리 흐름은 이해하고 있고, JSP 문법은 추가 학습하겠습니다"가 적절합니다.
- **Elasticsearch**: 경험 없음. 언급 불필요, 질문받으면 "관심 있게 보고 있는 기술"로 처리.

**언급 불필요한 부분**

- Elastic Search 경험 유무 (없어도 자격요건 아님)
- IDC 인프라 경험 (신입 수준에서 기대 안 함)
- Kotlin (앱 파트 전용)

## 2. 예상 질문 10개 + 모범 답변 포인트

**Q1. 실무 경험 없이 포트폴리오만 있는데, 실제 팀 개발 환경에서 적응할 수 있을지 걱정됩니다.**

- 면접관 의도: 가장 직접적인 약점 검증. 혼자 짠 코드와 협업 코드는 다르다는 걸 알고 있는지 확인
- 핵심 포인트: Plogging(4인 팀 백엔드 리드), FormCheck(5인 팀 백엔드 리드) 경험을 꺼내야 합니다. "코드 리뷰를 통해 피드백 주고받는 문화를 직접 만들어봤고, ADR로 의사결정을 문서화하는 습관이 있어 팀 맥락 공유에 자신 있습니다"라는 방향으로 답합니다.
- 피해야 할 패턴: "혼자서도 열심히 했으니 괜찮을 것 같습니다" 식의 막연한 자신감. 협업 경험의 구체적 사례 없이 끝내는 것.

---

**Q2. Redis Sorted Set으로 랭킹 조회를 99% 개선했다고 했는데, Sorted Set을 선택한 이유와 그 한계는 무엇인가요?**

- 면접관 의도: 기술 선택의 근거를 알고 있는지, 아니면 결과만 아는지 검증. 오토위니의 경매 랭킹 문제에 직접 연결되는 질문
- 핵심 포인트: 커버링 인덱스(I/O 감소하나 집계+정렬 제거 불가), Redis String 캐싱(TTL 만료 시 cache stampede 문제)을 거쳐 Sorted Set을 선택한 3단계 트레이드오프 과정을 설명합니다. 한계로는 "메모리 사용량 증가"와 "DB 원본 데이터와의 정합성 보장 문제"를 언급하면 됩니다.
- 피해야 할 패턴: "빠르니까 Redis를 썼습니다"로 끝내는 것. 한계를 모르는 척하는 것.

---

**Q3. JSP 경험이 없는데, 우리 서비스에는 JSP 기반 코드가 있습니다. 어떻게 하시겠습니까?**

- 면접관 의도: 레거시 코드에 대한 태도 확인. 도망가지 않고 직면할 수 있는 사람인지
- 핵심 포인트: "Servlet의 요청 처리 흐름은 이해하고 있습니다. Plogging 프로젝트에서 JSP/Servlet 기반 흐름을 직접 다뤄봤고, JSTL 문법은 추가로 숙지하겠습니다. 레거시 코드를 무작정 리팩토링하기보다 먼저 이해하고 최소한의 수정으로 접근하는 게 중요하다고 생각합니다"
- 피해야 할 패턴: "JSP는 구식이니 빨리 마이그레이션해야 한다"는 식으로 현재 레거시를 깎아내리는 것.

---

**Q4. 복합 인덱스 (category, created_at DESC)를 선택한 이유를 설명해주세요. 순서가 반대였다면 어떻게 됐을까요?**

- 면접관 의도: 인덱스 원리를 실제로 이해하는지 검증. 포트폴리오에 "팔로우업" 질문으로 직접 써있는 내용이라 면접관이 읽으면 반드시 물어볼 수 있습니다
- 핵심 포인트: WHERE 조건(category)이 선행 컬럼이어야 인덱스 레인지 스캔이 가능하고, ORDER BY(created_at)가 후행 컬럼으로 따라오면 filesort가 제거됩니다. (created_at, category)였다면 created_at 범위가 WHERE에 없어서 인덱스 효율이 떨어지고, category 필터링은 다시 filesort가 필요했을 것입니다.
- 피해야 할 패턴: "EXPLAIN으로 확인했더니 이게 더 빨랐습니다"로 측정 결과만 말하고 원리 설명을 못하는 것.

---

**Q5. Spring Security를 쓰셨는데, JWT 인증/인가 흐름을 설명해주세요.**

- 면접관 의도: Spring Security 실제 구현 이해도. MMT에서 실제로 사용한 기술이라 구체적으로 물어볼 것
- 핵심 포인트: Filter Chain에서 JwtAuthenticationFilter가 토큰을 파싱하고, SecurityContext에 Authentication 객체를 저장하는 흐름을 설명합니다. AccessToken/RefreshToken 분리 전략, 토큰 탈취 대응(짧은 만료시간 + RefreshToken 갱신)까지 연결하면 좋습니다.
- 피해야 할 패턴: "스프링 시큐리티가 알아서 처리해줍니다"식의 추상적 설명.

---

**Q6. GitHub Actions로 CI/CD를 구축하셨는데, 배포 중 장애가 발생하면 어떻게 대응하시겠습니까?**

- 면접관 의도: DevOps 문화 정착 의지 + 운영 경험 확인
- 핵심 포인트: 현재 구성(Docker 이미지 빌드 → Hub 푸시 → EC2 pull 재시작)에서 롤백은 이전 이미지 태그로 재배포하는 방식입니다. "지금 구성의 한계로 무중단 배포가 안 된다는 것을 알고 있고, Blue-Green 배포나 Nginx reload 전략을 팔로우업 과제로 보고 있습니다"라고 자기 인식을 보여주면 됩니다.
- 피해야 할 패턴: 롤백 방법을 모르는 것, 또는 "배포 중 장애는 없었습니다"로 회피하는 것.

**Q7. 대용량 데이터 처리 경험이 있다고 하셨는데, 10만 건과 실제 수백만 건 이상의 트래픽 처리는 차이가 있습니다. 어떻게 생각하시나요?**

- 면접관 의도: 경험의 한계를 솔직하게 인지하는지 확인. 자기 과신하는 사람을 걸러내는 질문
- 핵심 포인트: "10만 건은 학습용 스케일이고, 실서비스의 수백만 건과는 다르다는 것을 알고 있습니다. 다만 EXPLAIN 진단 → 병목 특정 → 전략 비교 → 측정 검증이라는 접근 방법론은 스케일이 커져도 동일하게 적용됩니다. 실서비스 레벨의 문제는 팀에서 배우며 적용해나가겠습니다"
- 피해야 할 패턴: "비슷할 것 같습니다" 식으로 차이를 모른 척하는 것.

**Q8. Neo4j에서 MySQL로 마이그레이션을 결정한 이유를 설명해주세요. 반대 의견이 있었다면 어떻게 설득했을 건가요?**

- 핵심 포인트: "그래프 데이터니까 그래프 DB"가 아니라 실제 쿼리 패턴이 단방향 조회뿐이었고, 이중 인프라 관리 비용이 이점을 초과한다는 판단 과정을 설명합니다. 1인 개발이라 설득 상대가 없었지만, "팀 개발이었다면 이 비교표를 기반으로 논의했을 것"이라고 자연스럽게 확장합니다.
- 피해야 할 패턴: "혼자 개발이라 그냥 결정했습니다"로 끝내는 것.

## 3. 프로젝트별 STAR 답변 초안

### Performance Tuning Project

**S (Situation)**
상품 10만 건, 주문 5만 건 규모의 이커머스 환경에서 카테고리별 최신순 상품 조회 시 Full Table Scan이 발생하고, 실시간 판매 랭킹 조회 시 매번 COUNT + ORDER BY가 실행되고 있었습니다. 이를 K6 부하 테스트로 측정한 결과, 상품 조회 p95가 70.6ms, 랭킹 조회 p95가 2,813ms였습니다.

**T (Task)**
각 병목의 원인을 EXPLAIN 실행계획으로 진단하고, 여러 전략을 비교해 최적의 해결책을 선택한 뒤 수치로 효과를 검증해야 했습니다.

**A (Action)**
상품 조회는 세 가지 인덱스 전략(단일 category 인덱스, 단일 created_at 인덱스, 복합 인덱스)을 실험해 (category, created_at DESC) 복합 인덱스가 WHERE + ORDER BY를 동시에 커버하는 유일한 구성임을 확인했습니다. 랭킹 조회는 커버링 인덱스(집계 제거 불가), Redis String 캐싱(cache stampede 문제), Redis Sorted Set(ZINCRBY 구조 저장) 세 가지를 단계적으로 시도했고, Sorted Set이 갱신 빈도가 높은 데이터는 결과가 아닌 정렬 구조 자체를 캐싱해야 한다는 원칙에 부합함을 확인했습니다.

**R (Result)**
상품 조회 p95 70.6ms → 34.3ms (51% 감소), 랭킹 조회 p95 2,813ms → 17ms (99.4% 감소). 인사이트: "성능 최적화는 추측이 아닌 측정"이라는 원칙을 체감했고, 이후 프로젝트에서는 EXPLAIN 진단을 먼저 수행하는 습관을 갖게 됐습니다.

### My Math Teacher

**S (Situation)**
수학 교사 시절 "학생이 틀린 문제의 근본 원인을 스스로 찾기 어렵다"는 문제를 직접 겪었습니다. AI로 취약 선수지식을 역추적하는 맞춤형 학습 플랫폼을 설계했고, 1인 개발로 기획부터 배포까지 전담했습니다.

**T (Task)**
서비스 로직뿐 아니라 5천 건 문항 테이블의 맞춤 추천 성능, 수작업 배포의 불안정성, TensorFlow Serving 연동까지 모든 기술적 문제를 직접 해결해야 했습니다.

**A (Action)**
맞춤 문항 추천의 ORDER BY RAND() 병목은 MMT 1.0의 시행착오를 반면교사로 삼아 EXPLAIN부터 진단했고, 인라인 뷰로 PK만 랜덤 선택하는 구조로 교체했습니다. 배포 불안정성은 GitHub Actions + Docker Compose CI/CD 파이프라인으로 자동화했습니다. Neo4j와 MySQL 이중 DB 유지 비용이 이점을 초과한다는 판단 하에 MySQL 단일 DB + 재귀 CTE로 마이그레이션하는 결정도 ADR로 문서화했습니다.

**R (Result)**
문항 추천 응답시간 232ms → 50ms (78% 감소), 배포 시간 25분 → 7분 (72% 감소). 인사이트: 교육 서비스는 기술 완성도보다 콘텐츠 품질이 서비스 성패에 결정적임을 체감했고, "기술이 문제를 해결할 수 있는가"보다 "해결할 가치가 있는 문제인가"를 먼저 검증하는 순서로 사고를 바꿨습니다.

## 4. 경력 전환 서사 포지셔닝

### 오토위니 맥락에서의 프레이밍

오토위니는 "유저가 겪는 문제를 기술로 해결하는 것에 관심 있는 사람"을 명시적으로 원합니다. 이 문구는 JD에서 굉장히 드문 표현입니다. 대부분의 JD가 기술 스택 숙련도를 요구하는 반면, 오토위니는 "문제 감지 능력"을 자격요건으로 넣었습니다. 이게 수학 교사 배경의 결정적 접점입니다.

**권장 서사 흐름**

> "수학 교사로 일하면서 '학생이 왜 이 문제를 틀렸는가'를 매일 분석했습니다. 단순히 오답을 고쳐주는 게 아니라 근본 원인이 되는 선수지식 부족을 찾아내는 것이 제 일이었습니다. 개발을 시작하면서 이 사고방식이 그대로 이어졌습니다. Performance Tuning 프로젝트에서도 '느리다'는 증상이 아니라 'Full Table Scan'이라는 원인을 EXPLAIN으로 찾아내고, 세 가지 전략을 비교한 뒤 수치로 검증했습니다. 교사 시절 매번 했던 진단→가설→검증의 구조가 개발에서도 동일하게 작동한다는 것을 체감하고 있습니다."

이렇게 말하면 수학 교사 경력이 "기술 경험 부족을 커버하는 스토리텔링"이 아니라 "지금 내가 가진 문제 해결 방식의 원천"으로 들립니다.

**약점으로 보이지 않는 추가 포인트**

실무 경험 없음 → "1인 개발 5개 프로젝트를 설계·구현·배포·운영까지 전담하며, 팀 개발에서라면 여러 명이 분담했을 의사결정을 혼자 했습니다. 그 결과 ADR 17건이라는 기록이 남았고, 팀 개발에서 이 의사결정 과정을 문서화하고 공유하는 것이 오히려 강점이 될 수 있다고 생각합니다."

### 꼬리 질문 대응법

**"그래서 개발자로 전향한 결정적 계기가 무엇인가요?"**
→ "수학 교사 시절 '이런 서비스가 있으면 좋겠다'는 생각을 자주 했는데, 직접 만들 수 없었습니다. MMT를 만들면서 기획부터 배포까지 혼자 결정하고 실행하는 과정이 즐거웠습니다. 특히 백엔드에서 복잡한 현실 문제를 논리적 구조로 추상화하는 과정이 수학적 모델링과 닮아 있다고 느꼈습니다."

**"수학 교사는 개인 사업자 성격이 강한데, 팀 개발 문화에 적응할 수 있나요?"**
→ "교직은 개인 수업이지만, 교육과정 설계와 학교 운영은 팀 합의로 이루어집니다. Plogging과 FormCheck에서 4~5인 팀 백엔드 리드를 맡으며 의견 충돌 상황에서도 기술적 근거로 설득하는 경험을 했습니다."

**"늦게 시작한 것에 대한 불안감은 없나요?"**
→ "있습니다. 그래서 포트폴리오에서 '무엇을 만들었나'가 아니라 '왜 그렇게 결정했나'를 보여주려 했습니다. 빠른 시간 안에 더 많은 경험을 쌓는 것보다, 각 결정에 대한 근거를 갖고 성장하는 게 더 가치 있다고 생각합니다."

## 5. 이 면접 형식에 맞는 추가 전략

### 1차 인터뷰 특이사항별 주의점

**코딩 테스트 (Java 1 + JS 2) 관련**

Java 문제는 아마 컬렉션 조작, 정렬, 문자열 처리 같은 기초 알고리즘이 나올 가능성이 높습니다. JS 2문제는 DOM 조작보다 배열/객체 처리, 비동기 콜백 패턴 같은 문제가 나올 수 있습니다. 6일 안에 "완벽한 알고리즘 실력"을 만드는 건 무리이므로, JS 기본 메서드(map, filter, reduce, Promise)와 Java Stream API를 빠르게 복습하는 데 집중하세요. 코테 점수보다 "풀이 과정을 말로 설명하는 능력"이 기술 면접에서 더 중요합니다.

**기술 면접 병행 가능성**

코테 결과와 무관하게 면접관이 코드를 보며 바로 기술 질문으로 전환할 수 있습니다. 코테 코드를 제출한 직후 "이 코드에서 개선할 수 있는 부분은요?"라고 물어볼 수 있으니, 본인 코드의 시간/공간 복잡도를 항상 말할 준비를 해두세요.

**레거시 + 신기술 공존 환경 대응**

Spring 5.x와 Spring Boot가 혼재하는 팀입니다. "레거시를 빨리 교체하자"는 인상보다 "기존 코드 맥락을 이해하면서 점진적으로 개선하겠다"는 태도가 훨씬 좋게 들립니다.
