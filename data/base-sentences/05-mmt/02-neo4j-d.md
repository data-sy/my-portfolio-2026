## 트레이드오프 맥락 (면접 대비용)

상황에서 핵심 제약 조건:
- 제약 1: 1인 개발 운영 — 인프라 관리 비용 최소화 필요
- 제약 2: 실제 쿼리 패턴 — 단방향 트리 탐색 (단원 → 선수 단원)
- 제약 3: MySQL 8.0 재귀 CTE 지원 — RDBMS로 트리 탐색 가능
- 제약 4: 개념 테이블 양쪽 사용 — 동기화 부담

최종 선택의 결정적 근거:
- Neo4j의 핵심 강점(Cypher 쿼리, 다방향 탐색)을 활용하지 않음
- MySQL 재귀 CTE로 동일 결과 구현 가능
- 인프라 이중 관리 + 데이터 동기화 제거가 장기적으로 더 큰 이점

만약 상황이 달랐다면:
- 다방향 그래프 탐색 필요 → Neo4j 유지
- 팀 운영 + 인프라 전담 → Neo4j 유지 부담 적음
- MySQL 5.x (CTE 미지원) → Neo4j 유지 또는 애플리케이션 레벨 구현

## neo4j-a 내용 병합 (꼬리질문 대비: "구체적으로 CTE는 어떻게 구현했나?")

MySQL 마이그레이션 후 재귀 CTE 구현 과정:

시도 1: Self Join (고정 깊이)
- parent_id 기반 Self Join으로 계층 조회. JOIN 횟수 = 탐색 깊이
- 한계: 깊이가 고정됨. 가변 깊이 탐색 불가

시도 2: 애플리케이션 레벨 반복 조회
- 부모 노드 조회 → 자식 노드 조회 → 반복
- 한계: 깊이만큼 DB 왕복 발생 (N번 쿼리)

시도 3: 재귀 CTE (WITH RECURSIVE)
- MySQL 8.0 WITH RECURSIVE로 가변 깊이 트리 탐색
- 단일 쿼리로 전체 하위 선수지식 조회
- 순환 참조 방지를 위해 최대 깊이 10 제한 설정

  WITH RECURSIVE prereq_tree AS (
    SELECT id, name, parent_id, 1 as depth
    FROM topics WHERE id = ?
    UNION ALL
    SELECT t.id, t.name, t.parent_id, pt.depth + 1
    FROM topics t
    JOIN prereq_tree pt ON t.parent_id = pt.id
    WHERE pt.depth < 10
  )
  SELECT * FROM prereq_tree;
