# Troubleshooting D: 트레이드오프형

## Neo4j → MySQL 마이그레이션 — 그래프 DB 도메인 적합성 트레이드오프 분석
| key | value |
|---|---|
| title | Neo4j → MySQL 마이그레이션 — 그래프 DB 도메인 적합성 트레이드오프 분석 |
| situation | 선수지식 그래프를 Neo4j로 구현했으나, 실제 사용 패턴이 그래프 DB의 강점(다방향 탐색, 가변 깊이)을 활용하지 못함. 인프라 복잡도 대비 이점 부족 |
| option_1_name | Neo4j 유지 |
| option_1_pros | 이미 구현 완료, 그래프 시각화 자연스러움, 향후 복잡한 그래프 쿼리 확장 가능 |
| option_1_cons | 인프라 이중 관리 (MySQL + Neo4j), 운영 비용 증가, 실제 쿼리 패턴은 단방향 트리 탐색뿐 |
| option_2_name | MySQL 마이그레이션 (재귀 CTE) |
| option_2_pros | 인프라 단순화 (MySQL 단일), 운영 비용 절감, 재귀 CTE로 트리 탐색 구현 가능 |
| option_2_cons | 그래프 특화 쿼리(최단 경로, 다방향 탐색) 포기, 마이그레이션 작업 필요 |
| option_3_name | 하이브리드 (MySQL 주DB + Neo4j 조회 전용) |
| option_3_pros | 데이터 정합성은 MySQL, 그래프 탐색은 Neo4j 활용 |
| option_3_cons | 데이터 동기화 복잡, 인프라 이중 관리 유지, 1인 개발 운영 부담 |
| final_choice | MySQL 마이그레이션 (재귀 CTE) |
| reasoning | 실제 쿼리 패턴이 "특정 노드의 하위 선수지식 탐색"으로 단방향 트리 구조. MySQL 8.0 재귀 CTE로 동일 성능 구현 가능. 1인 운영에서 인프라 단순화가 우선 |
| insight_1 | 기술 선택은 일회성이 아니라 도메인 적합성에 따라 재평가가 필요함. "그래프 데이터니까 그래프 DB"가 아니라, 실제 쿼리 패턴과 운영 비용을 종합적으로 판단해야 함 |
| insight_2 | "이미 구현했으니 유지하자"는 매몰 비용 오류. 현재 상황에서 최선의 선택을 다시 하는 것이 장기적으로 더 나은 결정 |
| followup_q1 | 만약 도메인이 소셜 네트워크처럼 복잡한 그래프였다면 다른 결정을 했을까? |
| followup_q2 | 마이그레이션 과정에서 데이터 정합성은 어떻게 검증했나? |

## 트레이드오프 맥락 (면접 대비용)

상황에서 핵심 제약 조건:
- 제약 1: 1인 개발 운영 — 인프라 관리 비용 최소화 필요
- 제약 2: 실제 쿼리 패턴 — 단방향 트리 탐색 (단원 → 선수 단원)
- 제약 3: MySQL 8.0 재귀 CTE 지원 — RDBMS로 트리 탐색 가능

각 선택지를 고려한 이유:
- Neo4j 유지: 이미 구현 완료, 변경 비용 0 → but 운영 비용 지속
- MySQL 마이그레이션: 인프라 단순화 → but 마이그레이션 작업 필요
- 하이브리드: 양쪽 장점 활용 → but 동기화 복잡도 증가

최종 선택의 결정적 근거:
- Neo4j의 핵심 강점(Cypher 쿼리, 다방향 탐색)을 활용하지 않음
- MySQL 재귀 CTE로 동일 결과 구현 가능
- 인프라 이중 관리 제거가 장기적으로 더 큰 이점

만약 상황이 달랐다면:
- 다방향 그래프 탐색 필요 → Neo4j 유지
- 팀 운영 + 인프라 전담 → 하이브리드 고려 가능
- MySQL 5.x (CTE 미지원) → Neo4j 유지 또는 애플리케이션 레벨 구현
