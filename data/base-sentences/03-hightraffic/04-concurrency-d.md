# Troubleshooting D: 트레이드오프형

## 선착순 쿠폰 동시성 제어 — 락 전략 트레이드오프 분석
| key | value |
|---|---|
| title | 선착순 쿠폰 동시성 제어 — 락 전략 트레이드오프 분석 |
| situation | 선착순 100명 쿠폰 발급 시 동시 1000명 요청으로 초과 발급 발생. 정합성을 보장하면서 성능 저하를 최소화할 락 전략 필요 |
| option_1_name | synchronized (JVM 락) |
| option_1_pros | 구현 단순, 추가 인프라 불필요 |
| option_1_cons | 단일 서버에서만 유효, Scale-out 시 무효 |
| option_2_name | DB 비관적 락 (SELECT FOR UPDATE) |
| option_2_pros | 분산 환경 지원, RDBMS만으로 구현 가능 |
| option_2_cons | 락 대기 시간만큼 커넥션 점유, 동시 1000명 시 커넥션 풀 고갈 위험 |
| option_3_name | Redis 분산락 + Lua Script |
| option_3_pros | DB 부하 없음, 원자적 수량 확인+차감, 분산 환경 완벽 지원 |
| option_3_cons | Redis 인프라 의존성 추가, 장애 시 복구 전략 필요 |
| final_choice | Redis 분산락 + Lua Script |
| reasoning | 다중 서버 환경에서 DB 커넥션 부담 없이 원자적 처리가 가능. 이미 캐시 용도로 Redis를 사용 중이므로 추가 인프라 비용 없음 |
| insight_1 | 동시성 제어는 "정합성 보장 수준 vs 처리 성능"의 트레이드오프. 락을 강하게 걸수록 정합성은 높아지지만 처리량이 떨어짐. 요구사항에 맞는 수준을 선택해야 함 |
| insight_2 | 기술 선택은 현재 인프라 맥락에 영향받음. Redis를 이미 사용 중이었기에 분산락 도입 비용이 낮았고, 그렇지 않았다면 DB 비관적 락이 더 현실적인 선택이었을 수 있음 |
| followup_q1 | Redis 장애 시 쿠폰 발급은 어떻게 되나? |
| followup_q2 | DB 비관적 락을 선택했다면 커넥션 풀 문제를 어떻게 완화했을까? |
