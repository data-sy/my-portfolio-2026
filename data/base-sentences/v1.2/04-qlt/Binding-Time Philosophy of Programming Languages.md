## 바인딩 시점으로 보는 언어 철학

언어 설계의 핵심 질문은 "무엇을 언제 확정하는가"

타입, 함수 호출, 메모리 해제, 의존성 주입, Null 체크 — 이런 결정들을 컴파일 타임에 확정할수록 안전하지만 유연성이 줄고, 런타임에 미룰수록 유연하지만 에러가 실행 중에 터진다. 이 결정의 총합이 언어의 철학을 결정한다.

### 바인딩이란?

"이 변수의 타입이 뭔지", "이 함수가 어떤 구현을 호출할지", "이 메모리를 언제 해제할지" 같은 결정들을 **언제 확정하느냐**가 바인딩 시점

| 결정 사항 | 컴파일 타임에 하면 | 런타임에 하면 |
|---|---|---|
| 타입 | 정적 타입 (Swift, Java) | 동적 타입 (JavaScript) |
| 함수 호출 | 정적 디스패치 (Swift) | 동적 디스패치 (Java virtual, JS) |
| 메모리 해제 | ARC (Swift), 수동 (C) | GC (Java, JavaScript) |
| 의존성 주입 | init 주입 (Swift) | 리플렉션 DI (Spring) |
| Null 체크 | Optional 강제 (Swift) | 런타임 NPE (Java), undefined (JS) |

- 컴파일에 확정할수록: 빠르고 안전하지만 유연성이 떨어짐
- 런타임에 확정할수록: 유연하지만 느리고 에러가 실행 중에 터짐

### 언어·프레임워크별 정적-동적 스펙트럼
```
컴파일 타임 ◀── C ── Swift ──────────── Java ──────────── JavaScript ──▶ 런타임
컴파일 타임 ◀─ SwiftUI ─────────────────────Spring─────────────────────▶ 런타임
```

### 세 언어의 탄생 배경과 철학

**Swift (2014 - 모바일 성능 목표)**
- 목표: Objective-C의 문제 해결 (런타임 에러, 메모리 관리 복잡성)
- 환경: 모바일 (배터리 제약, 메모리 제약, 60fps 필수, 앱 크래시 용납 불가)
- 철학: 컴파일 타임 안전성 > 개발자 편의성, 성능 예측 가능성
- 설계 결정:
  - Optional 강제: 앱 크래시 원천 차단 (Haskell의 Maybe 타입 영향)
  - ARC: GC pause 없이 성능 예측 가능, 컴파일 타임에 retain/release 삽입
  - Value Type: 멀티코어 시대, 공유 상태 문제 원천 차단, 함수형 패러다임
  - POP: 상속의 문제점 인식 (Fragile Base Class Problem, Composition over Inheritance)

**Java (1995 - 서버 생태계 목표)**
- 목표: "Write Once, Run Anywhere" (플랫폼 독립성)
- 환경: 기업용 서버 애플리케이션 (항상 켜져 있음, GC pause 감수 가능)
- 철학: 개발자 편의성 > 성능 예측성, 런타임 유연성으로 프레임워크 생태계 지원
- 설계 결정:
  - GC: 개발자가 메모리 걱정 안 해도 됨, 서버는 pause 감수 가능
  - null 허용: 1990년대엔 "null 안전성" 개념 자체가 희미, 런타임 유연성 우선
  - Reference Type: 객체지향의 정석 (Smalltalk 영향), 객체 공유로 효율성
  - OOP: 당시 주류 패러다임, 상속 기반 재사용

**JavaScript (1995 - 브라우저 스크립팅 목표)**
- 목표: 비개발자도 웹에 동적 기능을 빠르게 추가 (10일 만에 설계)
- 환경: 브라우저 (빠른 프로토타이핑, 유연한 DOM 조작, 다양한 입력 처리)
- 철학: 최대 유연성, 에러보다 실행 우선, 암묵적 타입 변환으로 "일단 돌아가게"
- 설계 결정:
  - 동적 타입: 선언 없이 바로 사용, 타입 변환 자동 처리
  - GC: Java와 동일하게 런타임 자동 수집
  - 프로토타입 상속: 클래스 없이 객체가 객체를 상속, 런타임에 구조 변경 가능
  - null + undefined: 두 가지 "없음"이 공존, 언어 레벨 안전장치 없음

**일관된 흐름:**
- Swift: 컴파일 안전성 우선 → 모든 설계 결정이 이를 지원 (Optional 강제, ARC, Value Type)
- Java: 중간 지점 → 정적 타입이지만 런타임 유연성도 확보 (null 허용, GC, Reference Type)
- JavaScript: 런타임 유연성 극대화 → 모든 결정을 최대한 늦춤 (동적 타입, 프로토타입, 암묵적 변환)

### 결론

- Swift의 강점: 컴파일 타임 안전성, 메모리 예측 가능성, 타입 안전성
- Java의 강점: 런타임 유연성, 풍부한 생태계, 프레임워크 편의성
- JavaScript의 강점: 최대 유연성, 빠른 프로토타이핑, 낮은 진입 장벽
- 공통점: "동작하는 코드"라는 목표는 동일, 안전성을 보장하는 시점과 방식이 다름
- 핵심: 탄생 배경(모바일 vs 서버 vs 브라우저)이 설계 철학을 결정 → 바인딩 시점이 달라짐

### 인사이트

언어마다 안전성을 보장하는 **시점**이 다름 (컴파일 vs 런타임).
Swift의 컴파일 타임 강제를 경험하니, Java의 런타임 유연성이 왜 필요한지 이해됨.
역으로 Java의 NPE 고통을 겪으니, Swift의 Optional 강제가 왜 존재하는지 납득.
JavaScript까지 보면 스펙트럼의 끝이 보임 — 런타임에 모든 걸 위임하면 TypeScript 같은 도구로 컴파일 타임을 다시 가져와야 하는 상황이 됨.
**환경이 다르면 최적해도 달라진다**는 걸 깨달음 (모바일은 60fps 필수 vs 서버는 GC pause 감수 vs 브라우저는 빠른 반복 우선).
새로운 언어를 배울 때도 "이 언어는 어떤 환경을 타겟으로 하고, 언제 무엇을 강제하나?"라는 관점으로 접근.
