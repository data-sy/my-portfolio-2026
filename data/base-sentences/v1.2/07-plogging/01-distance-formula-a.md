## 문제 해결 맥락 (면접 대비용)

상황: 현재 위치 기반 반경 1km 내 플로깅 장소 조회
SELECT * FROM places WHERE 거리(현재위치, 장소위치) <= 1.0;
→ 거리 계산 공식의 정확도가 결과 품질을 좌우
→ Java 8, JSP/Servlet 환경. DBMS Spatial 기능 미인지 상태

사고 흐름 (각 시도가 다음 시도의 근거):

시도 1: BETWEEN (사각형 필터링)
- 생각: "위도·경도 범위로 필터링하면 되지 않을까?"
- 방식: BETWEEN으로 사각형 범위 조회
- 한계: 사각형이라 원형 반경과 안 맞음. 초과 영역 27.3%
- 배운 것: 사각형 ≠ 원. 면적 문제

시도 2: 유클리드 거리 (평면 근사)
- 생각: "사각형 대신 원형으로 거리를 계산하면?"
- 방식: POWER 함수로 평면 유클리드 거리 계산
- 결과: 면적은 πr²로 초과 영역 문제 해결
- 한계: 면적은 맞지만 방향별 거리가 균일하지 않음. 경도 방향 최대 260m 과대평가
- 배운 것: 면적이 같아도 형태가 다르면 정확하지 않음. 위도·경도를 평면 좌표처럼 다루면 안 됨

시도 3: Haversine 공식 (구면 기하학)
- 생각: "위도·경도가 구면 좌표라면 구면 거리 공식을 써야"
- 방식: Math.sin, Math.cos, Math.atan2로 Java에서 Haversine 구현
- 결과: 최대 거리 오차 260m → 2.4m

인사이트 연결:
"좌표 두 개의 거리를 구하라"는 단순한 문제였지만,
사각형→원(면적 문제)과 평면→구면(형태 문제)이라는 두 가지 독립적인 문제를 발견.

이후 MySQL Spatial 기능을 알게 되면서, 기술 조사를 먼저 했다면
직접 구현 비용을 줄일 수 있었음을 깨달음.
→ 이 교훈은 이후 프로젝트에서 기술 선택 전 조사를 먼저 하는 습관으로 이어짐.
