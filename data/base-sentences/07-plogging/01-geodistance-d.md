# Troubleshooting D: 트레이드오프형

## 거리 계산 정확도 개선 — 성능 vs 정확도 트레이드오프 분석
| key | value |
|---|---|
| title | 거리 계산 정확도 개선 — 성능 vs 정확도 트레이드오프 분석 |
| situation | 반경 조회 시 거리 계산 오차율 32% 발생. 평면 좌표 기반 유클리드 거리 공식 사용 중. 성능과 정확도 중 어느 쪽을 우선할 것인가 판단 필요 |
| option_1_name | 유클리드 거리 (평면 근사) |
| option_1_pros | 계산 단순, 성능 우수 (사칙연산만) |
| option_1_cons | 지구 곡률 무시로 오차율 32%. 반경이 커질수록 오차 급증 |
| option_2_name | Haversine 공식 (구면 기하학) |
| option_2_pros | 지구 곡률 반영, 오차율 3.69%로 대폭 개선 |
| option_2_cons | 삼각함수 연산으로 유클리드 대비 계산 비용 증가 |
| option_3_name | Vincenty 공식 (타원체 기하학) |
| option_3_pros | 지구 타원체까지 반영, 최고 정확도 (오차 0.5mm 이하) |
| option_3_cons | 반복 계산(iteration) 필요, 구현 복잡도 높음, 위치 서비스에 과잉 정밀도 |
| final_choice | Haversine 공식 (구면 기하학) |
| reasoning | 플로깅 장소 반경 조회에서 수십 m 오차는 허용 불가(32% → 잘못된 장소 표시), 하지만 mm 단위 정밀도는 불필요. Haversine의 3.69% 오차는 위치 서비스에 충분하며 구현 복잡도도 합리적 |
| insight_1 | 트레이드오프 판단은 "요구사항 맥락"에서 시작해야 함. 위치 기반 서비스에서 "정확도 우선"은 "완벽한 정확도"가 아니라 "사용자가 체감하지 못할 수준의 오차"를 의미 |
| insight_2 | 이후 MySQL Spatial 데이터 타입과 ST_Distance_Sphere 함수의 존재를 학습. DBMS가 이미 제공하는 기능을 먼저 조사했다면 직접 구현 비용을 줄일 수 있었음 |
| followup_q1 | MySQL Spatial을 알았다면 다르게 구현했을까? |
| followup_q2 | 오차율 3.69%는 어떻게 측정했나? |

## 트레이드오프 맥락 (면접 대비용)

상황에서 핵심 제약 조건:
- 제약 1: 반경 조회 — 사용자 현재 위치 기준 주변 플로깅 장소 표시
- 제약 2: 정확도 — 잘못된 장소가 표시되면 사용자 신뢰 하락
- 제약 3: Java 8, JSP/Servlet — DBMS Spatial 기능 미인지 상태

각 선택지를 고려한 이유:
- 유클리드: 기존 구현, 가장 단순 → but 오차율 32%로 서비스 품질 부적합
- Haversine: 지구 곡률 반영 → 오차 3.69%로 충분한 정확도
- Vincenty: 최고 정확도 → but 위치 서비스에 과잉, 구현 복잡도 높음

최종 선택의 결정적 근거:
- 32% 오차는 "반경 1km 조회 시 320m 오차" → 사용자 체감 가능 수준
- 3.69% 오차는 "반경 1km 조회 시 37m 오차" → 사용자 체감 불가 수준
- Vincenty의 추가 정밀도는 이 서비스에서 가치 없음

만약 상황이 달랐다면:
- 측량/항법 서비스 → Vincenty 필수
- 전국 단위 대규모 조회 → MySQL Spatial + 공간 인덱스
- 성능 극한 최적화 필요 → 유클리드 + 사후 Haversine 필터링 (2단계)
