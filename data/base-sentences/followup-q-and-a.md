## DB 인덱스 최적화로 상품 목록 조회 성능 51% 개선
> 출처: `01-hightraffic/01-index-a.md`

**Q1.** 인덱스 컬럼 순서가 (created_at, category)였다면 어떻게 됐을까?

**A1-기존.** 실제로 순서를 바꿔서 EXPLAIN 확인한 결과, created_at이 선행 컬럼이면 범위 검색 시 후행 컬럼(category)의 필터링이 인덱스를 타지 못함. B-Tree 특성상 선행 컬럼이 범위 조건이면 후행 컬럼의 정렬/필터 이점을 잃기 때문에 (category, created_at) 순서가 적합.

**A1-생성.** 결론부터 말씀드리면, (created_at, category) 순서였다면 인덱스 효과가 크게 감소합니다. B-Tree 인덱스는 선행 컬럼이 범위 조건(ORDER BY DESC)일 때 후행 컬럼의 동등 조건(WHERE category = ?)을 인덱스로 처리하지 못합니다. 즉 created_at으로 정렬은 가능하지만 category 필터링에서 인덱스를 타지 못해 filesort가 다시 발생합니다. 이 프로젝트에서는 실제로 순서를 바꿔 EXPLAIN을 재실행하여 이를 검증했고, "동등 조건 컬럼을 선행, 범위/정렬 컬럼을 후행"이라는 복합 인덱스 설계 원칙을 체득했습니다.

**Q2.** 복합 인덱스를 사용했을 때 쓰기 비용이 늘어나지는 않나?

**A2-기존.** 상품 조회가 등록보다 압도적으로 많은 읽기 중심 서비스라 읽기 최적화 우선. 쓰기 빈도가 높아지면 인덱스 개수 조정이나 비동기 처리 고려.

**A2-생성.** 복합 인덱스를 추가하면 INSERT/UPDATE 시 인덱스 B-Tree를 갱신해야 하므로 쓰기 비용이 증가하는 것은 맞습니다. 하지만 이 서비스는 상품 등록보다 조회가 압도적으로 많은 읽기 중심 워크로드이기 때문에 읽기 최적화를 우선했습니다. 만약 쓰기 빈도가 높아진다면 불필요한 인덱스를 정리하거나, 쓰기 작업을 비동기 큐로 처리하는 방향을 검토할 것입니다. 핵심은 워크로드 특성에 따라 읽기/쓰기 트레이드오프를 판단하는 것입니다.

## Redis Sorted Set으로 실시간 랭킹 조회 98% 개선
> 출처: `01-hightraffic/02-ranking-a.md`

**Q1.** Sorted Set의 메모리 사용량이 커지면 어떻게 관리하나?

**A1-기존.** Sorted Set은 member 수에 비례하여 메모리 사용. TOP 100만 필요하므로 ZREMRANGEBYRANK로 하위 데이터 주기적 정리. 또는 TTL 설정으로 전체 키 만료 후 재구축.

**A1-생성.** Sorted Set의 메모리는 member 수에 비례하므로, 실제로 필요한 것은 TOP 100뿐이라는 점을 활용합니다. ZREMRANGEBYRANK 명령어로 순위 하위의 데이터를 주기적으로 정리하면 메모리를 일정 수준으로 유지할 수 있습니다. 또한 키 자체에 TTL을 설정해 일정 주기로 만료 후 DB 기반으로 재구축하는 전략도 가능합니다. 상품 15만 건 기준으로 Sorted Set은 약 수 MB 수준이라 현재 규모에서는 문제가 되지 않지만, 규모가 커질 경우를 대비한 정리 정책을 미리 설계해두는 것이 중요합니다.

**Q2.** 랭킹 데이터와 DB 원본 데이터의 정합성은 어떻게 보장하나?

**A2-기존.** 판매 발생 시 DB 업데이트와 동시에 ZINCRBY로 Redis 갱신. 장애 시 DB 기준 배치로 Redis 재구축하는 복구 전략 마련.

**A2-생성.** 판매 발생 시점에 DB 주문 테이블 INSERT와 Redis ZINCRBY를 함께 수행하여 실시간 정합성을 유지합니다. 다만 Redis 장애나 네트워크 이슈로 불일치가 발생할 수 있으므로, DB의 주문 데이터를 기준으로 Sorted Set을 재구축하는 배치 작업을 복구 전략으로 마련했습니다. 랭킹은 "정확한 실시간"보다 "충분히 빠른 근사치"가 중요한 도메인이라, 순간적 불일치는 허용하되 DB를 Single Source of Truth로 두고 복구 가능한 구조를 설계한 것입니다.

## 인라인 뷰 최적화로 맞춤 문항 API 랜덤 조회 78% 개선
> 출처: `02-mmt/01-randomquery-a.md`

**Q1.** EXPLAIN을 사용했다면 어떤 병목을 발견했을까?

**A1-기존.** filesort + Using temporary를 확인했을 것. ORDER BY RAND()는 전체 행에 난수를 부여한 후 정렬하기 때문에 임시 테이블 생성 및 정렬 비용이 발생. 인라인 뷰로 PK만 대상으로 하면 정렬 대상이 줄어들고, 인덱스만으로 처리 가능하다는 점을 즉시 파악할 수 있었음.

**A1-생성.** EXPLAIN을 실행했다면 Extra 컬럼에서 Using temporary와 Using filesort를 즉시 확인했을 것입니다. ORDER BY RAND()는 조건에 맞는 전체 행에 난수를 부여하고 임시 테이블에 저장한 뒤 정렬하므로, 두 가지 병목이 동시에 발생합니다. 이를 보면 "정렬 대상 자체를 줄여야 한다"는 방향을 바로 잡을 수 있었고, PK만 대상으로 서브쿼리에서 랜덤 선택하는 인라인 뷰 최적화로 직행할 수 있었을 것입니다. 이 경험에서 "EXPLAIN 진단을 먼저 수행하자"는 교훈을 얻었고, 이후 HT 프로젝트에서 실제로 적용하여 시행착오 없이 문제를 해결했습니다.

**Q2.** ORDER BY RAND()가 항상 나쁜 선택인가?

**A2-기존.** 데이터가 수백~수천 건이면 ORDER BY RAND()도 허용 가능. 문제는 수만 건 이상에서 발생. 데이터 규모와 조회 빈도에 따라 판단해야 하며, 무조건 피하기보다 상황에 따라 결정.

**A2-생성.** 항상 나쁜 것은 아닙니다. 데이터가 수백~수천 건 수준이고 조회 빈도가 낮다면 ORDER BY RAND()의 성능 저하는 체감되지 않으며, 구현이 가장 간결하다는 장점이 있습니다. 문제가 되는 것은 수만 건 이상에서 빈번하게 호출되는 경우입니다. 이 프로젝트에서는 문항 5천 건에 500명 동시 부하라는 조건이었기 때문에 최적화가 필요했습니다. 기술 선택은 "절대적으로 나쁜 방법"이 아니라 데이터 규모와 트래픽 패턴에 따라 판단해야 합니다.

## Neo4j -> MySQL 마이그레이션 -- DBMS 선택의 트레이드오프
> 출처: `02-mmt/02-neo4j-d.md`

**Q1.** 팀 개발이었다면 같은 선택을 했을까?

**A1-기존.** 팀에 인프라 전담 인력이 있고 Neo4j 운영 부담이 분산된다면 유지했을 수도 있음. 하지만 여전히 "실제 쿼리 패턴이 Neo4j의 강점을 활용하지 못한다"는 핵심 문제는 남아있음. 소셜 네트워크, 추천 시스템, 부정탐지 등 다방향 + 가변 깊이 탐색이 핵심인 도메인이면 Neo4j 유지가 합리적. 이 프로젝트는 "특정 단원 -> 선수 단원" 단방향 트리 탐색이라 RDBMS로 충분.

**A1-생성.** 팀에 인프라 전담 인력이 있다면 Neo4j 운영 부담이 분산되므로 유지를 검토할 여지는 있습니다. 하지만 핵심 판단 기준은 "실제 쿼리 패턴이 Neo4j의 강점을 활용하는가"이고, 이 프로젝트는 단방향 트리 탐색만 사용했기 때문에 팀 규모와 무관하게 MySQL이 적합합니다. Neo4j가 빛나는 것은 소셜 네트워크의 친구 추천, 부정탐지의 다방향 패턴 탐색처럼 그래프 특화 쿼리가 핵심인 도메인입니다. 팀이었다면 ADR(Architecture Decision Record)을 작성하여 판단 근거를 문서화하고 팀 합의를 거쳤을 것입니다.

**Q2.** MySQL로 마이그레이션했을 때 성능이 느려지지는 않았나?

**A2-기존.** 단방향 트리 탐색(깊이 5~7) 기준 Neo4j Cypher와 MySQL 재귀 CTE 모두 수십 ms 내로 유사한 성능. 이 프로젝트의 데이터 규모(수천 노드)에서는 체감 차이 없음. Neo4j의 인덱스 프리 인접성(Index-Free Adjacency)은 수백만 노드 + 다방향 탐색에서 이점이 드러나는 특성.

**A2-생성.** 단방향 트리 탐색(깊이 5~7) 기준으로 Neo4j Cypher와 MySQL 재귀 CTE 모두 수십 ms 이내로 유사한 성능이었고, 수천 노드 규모에서 체감 차이는 없었습니다. Neo4j의 핵심 강점인 인덱스 프리 인접성(Index-Free Adjacency)은 수백만 노드 규모에서 다방향 탐색을 할 때 비로소 이점이 드러나는 특성입니다. 이 프로젝트의 규모와 쿼리 패턴에서는 MySQL 재귀 CTE로 충분했고, 오히려 인프라 단일화와 동기화 문제 제거라는 운영상 이점이 더 컸습니다.

## GitHub Actions + Docker Compose로 배포 시간 72% 단축
> 출처: `02-mmt/03-deploy-e.md`

**Q1.** 배포 중 장애가 발생하면 롤백은 어떻게 하나?

**A1-기존.** Docker 이미지에 태그(버전)를 부여하여 이전 버전 이미지로 docker-compose up 실행. 추가로 DB 마이그레이션이 포함된 배포는 롤백이 복잡하므로 배포 전 백업 스크립트 실행.

**A1-생성.** Docker 이미지에 빌드마다 버전 태그를 부여하고 있어서, 장애 발생 시 이전 버전 이미지로 docker-compose up을 실행하면 즉시 롤백이 가능합니다. 다만 DB 스키마 마이그레이션이 포함된 배포는 단순 이미지 롤백으로 해결되지 않으므로, 배포 전에 DB 백업 스크립트를 실행하는 절차를 추가했습니다. 소규모 빈번한 배포가 가능해진 것도 롤백에 유리한데, 변경 범위가 작을수록 원인 파악과 복구가 빠르기 때문입니다.

**Q2.** GitHub Actions의 비용이나 제약은 없었나?

**A2-기존.** GitHub Actions는 공개 레포 무료, 비공개 레포는 월 2,000분 무료 제공. 빌드 시간이 7분이므로 월 280회 이상 배포 가능하여 개인 프로젝트에는 충분. Self-hosted runner도 고려했으나 관리 비용 대비 이점 없어 GitHub-hosted 사용.

**A2-생성.** GitHub Actions는 비공개 레포 기준 월 2,000분 무료로, 빌드 시간 7분 기준 월 280회 이상 배포가 가능하여 개인 프로젝트에는 충분했습니다. Self-hosted runner도 검토했으나, 별도 서버 관리 비용 대비 이점이 없어 GitHub-hosted runner를 사용했습니다. 제약이라면 러너 스펙이 고정되어 빌드 속도 튜닝에 한계가 있다는 점인데, Gradle 빌드 캐시와 Docker 멀티스테이지 빌드로 충분히 대응할 수 있었습니다.

## 좌표계 특성을 반영한 위치 기반 거리 계산 공식 선택
> 출처: `03-plogging/01-distance-formula-a.md`

**Q1.** Oracle Spatial 기능을 알았다면 어떻게 구현했을까?

**A1-기존.** MySQL Spatial 기능(또는 Oracle Spatial)을 알았다면 ST_Distance_Sphere 함수를 사용했을 것. POINT 타입으로 좌표를 저장하고 공간 인덱스를 생성하여 DB 레벨에서 거리 계산 + 공간 인덱스 기반 필터링이 가능. 애플리케이션 레벨에서 계산하는 것보다 성능이 우수하고 코드도 간결해짐.

**A1-생성.** Spatial 기능을 알았다면 좌표를 POINT 타입으로 저장하고, 공간 인덱스(R-Tree)를 생성한 뒤, ST_Distance_Sphere 함수로 DB 레벨에서 구면 거리 계산과 필터링을 한 번에 처리했을 것입니다. 애플리케이션에서 Haversine을 직접 구현하는 것보다 코드가 간결하고, 공간 인덱스 덕분에 대량 데이터에서도 성능이 우수합니다. 다만 직접 구현한 경험이 있었기에 ST_Distance_Sphere가 내부적으로 Haversine 기반 계산을 수행한다는 것을 이해하고 있고, 이 경험에서 "데이터 특성에 특화된 DBMS 기능을 우선 검토해야 한다"는 교훈을 얻었습니다.

**Q2.** Spatial 기능을 쓰면 Java에서 Haversine 직접 구현하는 것보다 어떤 이점이 있을까?

**A2-기존.** (1) DB 레벨 필터링으로 네트워크 전송 데이터 감소. (2) 공간 인덱스(R-Tree) 활용으로 대량 데이터에서 쿼리 성능 향상. (3) 검증된 구현체로 정확도 보장. 하지만 직접 구현 경험 덕분에 Haversine 공식의 원리와 Spatial 함수의 내부 동작을 깊이 이해하게 됨. 기술을 선택하더라도 그 내부를 이해하고 있으면 디버깅과 최적화에 유리.

**A2-생성.** 크게 세 가지 이점이 있습니다. 첫째, DB 레벨에서 필터링하므로 불필요한 데이터가 애플리케이션까지 전송되지 않아 네트워크 비용이 줄어듭니다. 둘째, R-Tree 공간 인덱스를 활용해 대량 데이터에서도 빠른 범위 검색이 가능합니다. 셋째, 검증된 구현체이므로 정확도를 별도로 검증할 필요가 없습니다. 다만 직접 Haversine을 구현한 경험이 있어서 Spatial 함수의 내부 동작 원리를 이해하고 있고, 이는 성능 이슈 디버깅이나 최적화 판단에서 강점이 됩니다.

## DB vs 애플리케이션 -- 거리 계산 처리 위치의 트레이드오프
> 출처: `03-plogging/02-processing-location-d.md`

**Q1.** Spatial 기능과 하이브리드 방식의 트레이드오프는?

**A1-기존.** MySQL Spatial 기능(POINT 타입 + ST_Distance_Sphere 함수 + 공간 인덱스)을 쓰면 DB 레벨에서 구면 거리 계산과 공간 인덱스(R-Tree) 기반 필터링이 모두 가능. 하이브리드보다 구현이 단순하고 대량 데이터에서 성능도 우수할 수 있음. 당시에는 Spatial 기능을 인지하지 못했고, 하이브리드 방식으로 충분한 결과를 확보함. 지금 다시 한다면 Spatial 기능을 우선 검토하되, 데이터 규모와 쿼리 빈도를 고려해 선택할 것.

**A1-생성.** Spatial 기능은 POINT 타입 + ST_Distance_Sphere + R-Tree 공간 인덱스로 DB 레벨에서 구면 거리 계산과 필터링을 모두 처리하므로 구현이 단순하고 대량 데이터에서 성능도 우수합니다. 반면 하이브리드 방식은 DB BETWEEN으로 인덱스를 활용한 1차 필터링 + Java Haversine 2차 필터링으로, Spatial 지원이 없는 환경에서도 적용 가능하다는 장점이 있습니다. 지금 다시 한다면 Spatial 기능을 우선 검토하되, DBMS 버전이나 인프라 제약이 있다면 하이브리드도 충분히 유효한 선택입니다. 이 경험에서 "기술 선택 전에 DBMS가 제공하는 특화 기능부터 조사해야 한다"는 교훈을 얻었습니다.

## SRP 기반 단계적 구조 개선
> 출처: `04-qlt/02-srp-layer-separation-a.md`

**Q1.** Service가 비대해지면 어떻게 분리하나?

**A1-기존.** 책임별로 분리. 현재 TimerCompletionHandler가 완료 처리를 분담. 알림도 NotificationService로 분리 가능. Service는 오케스트레이션만 담당하는 Facade 패턴으로 진화.

**A1-생성.** 책임별로 분리하는 것이 원칙입니다. 현재도 TimerCompletionHandler가 완료 후처리를 분담하고 있고, 알림 관련 로직은 NotificationService로 추가 분리할 수 있습니다. Service 자체는 비즈니스 흐름을 조율하는 오케스트레이션 역할(Facade 패턴)만 담당하고, 실제 처리는 각 전문 객체에 위임하는 구조로 진화시킵니다. Spring의 @Service도 동일한 원칙으로, Service가 비대해지면 도메인 서비스를 분리하는 것과 같은 접근입니다.

**Q2.** 이 계층 분리가 Spring 환경과 어떻게 연결되나?

**A2-기존.** Spring의 Controller->Service->Repository 계층과 동일한 구조. @Service는 비즈니스 로직 집중, @Transactional은 작업 원자성 보장. 이 Service도 "데이터 수정 + 부수 효과"의 원자성을 보장하는 동일한 역할. Spring 경험이 있었기에 Swift에서도 자연스럽게 계층 분리를 적용할 수 있었음.

**A2-생성.** Swift에서 적용한 View->ViewModel->Service->Repository 구조는 Spring의 Controller->Service->Repository 계층과 동일한 원리입니다. Spring의 @Service가 비즈니스 로직에 집중하고 @Transactional이 작업 원자성을 보장하듯, 이 프로젝트의 Service도 "데이터 수정 + 부수 효과(알림 스케줄링)"의 원자성을 보장합니다. Spring 경험이 있었기에 Swift에서도 자연스럽게 계층 분리를 적용할 수 있었고, 역으로 Swift에서 직접 계층을 설계해본 경험이 Spring의 각 레이어가 왜 그렇게 나뉘었는지를 더 깊이 이해하게 해주었습니다.

## Java vs Swift 언어 철학 비교
> 출처: `04-qlt/03-java-swift-b.md`

**Q1.** JavaScript처럼 런타임 중심 언어를 써봤다면, 타입 안전성을 어떻게 보완할까?

**A1-기존.** JavaScript(Node.js)를 쓴다면 TypeScript 도입이 첫 번째 선택. 컴파일 타임에 타입 체크를 추가해서 Swift처럼 안전성을 앞당기는 효과. 다만 TypeScript도 런타임에는 타입이 사라지므로 Zod 같은 런타임 검증 라이브러리를 병행해야 완전한 안전성 확보. 결국 언어가 보장 안 하는 부분은 도구로 보완하는 것이고, Swift->Java->JavaScript 순으로 "개발자가 직접 챙겨야 할 안전성"이 늘어나는 구조

**A1-생성.** 첫 번째로 TypeScript를 도입하여 컴파일 타임 타입 체크를 추가할 것입니다. Swift처럼 언어 레벨에서 안전성을 앞당기는 효과가 있습니다. 다만 TypeScript는 런타임에 타입이 사라지므로 외부 입력(API 요청, 사용자 입력) 경계에서는 Zod 같은 런타임 검증 라이브러리를 병행해야 합니다. Swift->Java->JavaScript 순으로 "개발자가 직접 챙겨야 할 안전성"이 늘어나는 스펙트럼이 있고, 언어가 보장하지 않는 부분은 도구와 컨벤션으로 보완하는 것이 핵심입니다.

**Q2.** 세 언어의 스펙트럼을 경험한 뒤, 새로운 언어를 평가할 때 가장 먼저 보는 기준은?

**A2-기존.** "이 언어는 안전성을 언제 보장하는가?"를 가장 먼저 봄. 컴파일 타임 강제가 많으면 초기 학습 비용은 높지만 런타임 버그가 줄고, 런타임 유연성이 높으면 빠르게 프로토타이핑 가능하지만 방어 코드가 늘어남. Swift->Java->JavaScript 경험으로 이 트레이드오프를 체감했기 때문에, 새 언어를 볼 때 Null 처리·타입 시스템·메모리 모델을 기준으로 스펙트럼 어디쯤인지 먼저 파악하고 학습 전략을 세움

**A2-생성.** "이 언어는 안전성을 언제 보장하는가?"를 가장 먼저 봅니다. 구체적으로 Null 처리 방식, 타입 시스템의 강도, 메모리 모델을 기준으로 컴파일 타임-런타임 스펙트럼 어디에 위치하는지 파악합니다. 컴파일 타임 강제가 많으면 초기 학습 비용은 높지만 런타임 버그가 줄고, 런타임 유연성이 높으면 빠른 프로토타이핑이 가능하지만 방어 코드가 늘어납니다. Swift, Java, JavaScript를 경험하며 이 트레이드오프를 체감했기 때문에, 새 언어를 접할 때 문법보다 먼저 설계 철학과 타겟 환경을 파악하고 학습 전략을 세웁니다.

**A2-생성.** MediaPipe는 2D 영상에서 3D 좌표를 추정하지만 z축(깊이) 정확도는 상대적으로 낮습니다. x-y 평면 위주 동작(덤벨 숄더프레스의 팔꿈치 굽힘 등)은 정확도에 문제가 없었지만, 시티드 로우처럼 팔이 몸 안쪽으로 당겨지는 동작은 z축 오차로 판정 정확도가 떨어졌습니다. 이 한계를 인지하고 카메라 배치 가이드(측면 촬영 권장)로 완화했으며, 근본적 해결은 웨어러블 IMU 센서 등 실제 3D 관절 움직임을 직접 측정하는 방향이 필요하다고 판단했습니다. 기술의 한계를 명확히 인식하고 대응 방안을 제시하는 것이 중요합니다.

## 실시간 자세 교정 서비스 개발 및 발표 -- 최종 발표회 최우수상 (12팀 중 2등)

**Q1.** 발표에서 기술적 깊이와 청중 이해 수준의 균형은 어떻게 맞췄나?

**A1-기존.** 발표 구성을 "문제 -> 해결 과정 -> 결과" 스토리로 설계. 기술 용어는 비유로 먼저 설명한 뒤 정확한 명칭 제시. 시연(데모)을 통해 "이 기술이 실제로 무엇을 하는지" 직관적으로 보여줌. 질의응답에서는 기술적 깊이로 답변.

**A1-생성.** 발표 본문은 "문제 -> 해결 과정 -> 결과" 스토리 구조로 설계하고, 기술 용어는 비유로 먼저 설명한 뒤 정확한 명칭을 제시했습니다. 예를 들어 구면좌표 변환은 "팔을 뻗은 방향을 숫자로 표현하는 방법"으로 먼저 설명한 뒤 기술적 배경을 덧붙였습니다. 데모를 통해 "이 기술이 실제로 무엇을 하는지" 직관적으로 보여주고, 질의응답에서는 기술적 깊이로 답변하는 전략을 취했습니다. 핵심은 발표와 Q&A에서 요구되는 깊이 수준이 다르다는 점을 인식한 것입니다.

**Q2.** 5인 팀에서의 역할 분담은 어떻게 이루어졌나?

**A2-기존.** 5인 팀: FE 2명 + BE 2명 + 기획/디자인 1명. 본인은 BE(AI 서버 + 자세 교정 로직) + 발표 담당. 역할 분담은 초기 기획 회의에서 각자 관심사와 강점 기반으로 결정. BE 간 역할은 운동별로 분담하되, 공통 모듈은 본인이 먼저 구현 후 팀 공유.

**A2-생성.** FE 2명, BE 2명, 기획/디자인 1명으로 구성되었고, 초기 기획 회의에서 각자의 관심사와 강점을 기반으로 역할을 결정했습니다. 저는 BE(Flask AI 서버 + MediaPipe 자세 교정 로직)와 발표를 담당했습니다. BE 간 역할은 운동별로 분담했는데, 첫 운동 구현 후 반복 패턴을 발견하여 공통 각도 계산 모듈을 만들고 팀에 공유했습니다. 그 결과 팀 전체의 운동별 구현시간이 3일에서 0.5일로 단축되었고, 개인 기여를 팀 전체 효율로 확장한 경험이었습니다.
