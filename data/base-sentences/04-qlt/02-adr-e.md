# Troubleshooting E: 일반형

## ADR로 의사결정 과정 체계적 기록 — 총 10건 작성
| key | value |
|---|---|
| title | ADR로 의사결정 과정 체계적 기록 — 총 10건 작성 |
| situation | 기능 추가 시 사이드이펙트 검토가 필요했으나, 의사결정 근거가 기록되지 않아 이전 판단을 되짚기 어려움 |
| action_1 | ADR(Architecture Decision Record) 형식 도입 — 상황/선택지/결정/근거 구조화 |
| action_2 | 기능 추가·변경마다 사이드이펙트 체크리스트 작성 |
| action_3 | 의사결정 시점에 ADR 작성, 총 10건 누적 |
| action_4 | 이후 유사 판단 시 기존 ADR 참조하여 일관성 유지 |
| action_5 | null |
| result | 총 10건의 ADR 작성. 기능 변경 시 이전 결정 근거를 빠르게 확인 가능. 사이드이펙트 사전 검토로 출시 후 결함 최소화 |
| insight_1 | 1인 개발에서도 의사결정 기록은 필수. "왜 이렇게 했지?"라는 질문은 본인에게도 발생하며, ADR은 미래의 나를 위한 문서 |
| insight_2 | ADR은 "정답을 기록"하는 것이 아니라 "판단 맥락을 기록"하는 것. 상황이 바뀌면 결정도 바뀔 수 있으며, 그때 기존 맥락을 아는 것이 중요 |
| followup_q1 | ADR의 구체적인 작성 포맷은? |
| followup_q2 | 10건 중 기존 결정을 번복한 사례가 있나? |

## 경험 맥락 (면접 대비용)

상황 배경:
- Quick Label Timer 개발 중 기능 추가·변경이 반복
- "이전에 왜 이렇게 했지?" 의문이 자주 발생
- 1인 개발이라 물어볼 팀원이 없음

액션 선택 이유:
- ADR 도입: 기술 블로그에서 학습, 간결한 형식이 1인 개발에 적합
- 사이드이펙트 체크리스트: 기능 간 의존성 파악에 도움
- 10건 누적: 앱 라이프사이클 전체에 걸쳐 지속적으로 작성

결과에서 의미 있는 부분:
- 정량적: ADR 10건 누적
- 정성적: 의사결정 속도 향상, 일관성 유지, 사이드이펙트 사전 차단

- ADR 템플릿을 프로젝트 초기부터 준비했으면 더 일관적이었을 것
- 코드 내 주석과 ADR의 연결(링크)을 더 체계적으로 관리

인사이트 연결:
ADR은 거창한 문서가 아니라 "왜?"에 대한 짧은 답변.
